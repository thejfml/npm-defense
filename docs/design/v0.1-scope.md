# v0.1 scope

The goal of v0.1 is a single-binary CLI that reads a lockfile, applies a fixed
set of detection rules, and prints a human-readable report. Nothing more.

## User story

> As a developer who just ran `npm install` or updated dependencies, I want to
> run `npm-defense scan` in my project directory and get a report that tells me
> whether any of my dependencies look suspicious, before I commit the lockfile
> change or ship to production.

## Success criteria

v0.1 is done when:

1. A fresh clone, `go build`, and `./npm-defense scan` in a real project
   directory produces a report in under 30 seconds for a project with ~500
   transitive dependencies (assuming warm cache).
2. The six rules in `rules.md` are implemented and each has at least one
   positive and one negative test case.
3. The axios@1.14.1 case (see `case-studies/axios-march-2026.md`) is caught
   with a high-severity finding, using a captured snapshot of the registry
   metadata as of the attack window.
4. Running the tool against a clean, up-to-date project produces zero false
   positives on at least three real-world projects (to be chosen — probably
   a small Vue app, an Express backend, and a TypeScript library).
5. There are no network calls except to `registry.npmjs.org`. Verified by
   test. No telemetry, no update checks, no error reporting.

## Non-goals for v0.1

- **No interactive mode.** Batch, one-shot only.
- **No config file.** All behavior via CLI flags. A config file can come in
  v0.2 once we know what's actually worth configuring.
- **No rule authoring by end users.** The rule set is fixed and compiled in.
  User-defined rules are a v0.3+ concern.
- **No SBOM output.** Report is human-readable (and optionally JSON for
  tooling). SBOM/SARIF formats come later if there's demand.
- **No auto-fix.** The tool reports; the human decides.
- **No integration with `npm audit` or GitHub Advisory DB.** That's a feed
  ingestion problem and explicitly deferred.

## CLI surface (draft — to be finalized in architecture.md)

```
npm-defense scan [path]            # scan project at path (default: .)
npm-defense scan --json            # machine-readable output
npm-defense scan --fail-on=high    # exit 1 if any finding >= threshold
npm-defense explain <rule-id>      # print what a rule does and why
npm-defense version
```

## Output sketch

```
npm-defense scan — 3 findings (1 high, 2 medium)

[HIGH]   axios@1.14.1
         New transitive dependency 'plain-crypto-js@4.2.1' added.
         plain-crypto-js was published 2h before this axios version.
         plain-crypto-js runs a postinstall script.
         Rules fired: R1 (dep-diff), R2 (package-age), R3 (install-script)

[MEDIUM] some-other-package@2.0.0
         Install script present: "postinstall": "node build.js"
         Rule fired: R3 (install-script)

[MEDIUM] typosquattish@1.0.0
         Name is within Levenshtein distance 2 of popular package 'typescript'
         Rule fired: R5 (typosquat)

Run `npm-defense explain R1` for details on any rule.
```

## Estimated effort

Rough, honest estimates assuming weekend work and AI-assisted coding:

- Project scaffolding, lockfile parsing, registry client, cache: 1 weekend.
- Rules R1–R4 with tests: 1 weekend.
- Rules R5, R6 with tests: 1 weekend.
- Report rendering, CLI polish, real-world project testing, README: 1 weekend.

Total: 4 weekends. If it takes 6, that's still fine. If it takes 10, something
is wrong — stop and reassess.
